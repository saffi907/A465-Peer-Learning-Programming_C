Vulnerability Recap

Stage 1 Information Disclosure
Vulnerability: A debug API endpoint (/api/debug) was left in production, leaking secret keys, database credentials, and internal routes.
Fix: Remove all debug/development endpoints before deployment. Store secrets in environment variables, never in code. Use feature flags to gate debug functionality.
Stage 2 Mass Assignment
Vulnerability: The registration handler blindly accepted all form fields, allowing an attacker to inject a role=admin parameter and escalate privileges.
Fix: Whitelist accepted input fields on the server. Only extract the specific fields you expect (username, password). Never trust client-submitted data.
Stage 3 Server-Side Template Injection
Vulnerability: User input was passed directly to render_template_string(), allowing Jinja2 template expressions to execute on the server and leak configuration secrets.
Fix: Never pass raw user input to a template engine. Use render_template() with separate template files. Escape all user input. Consider sandboxing the template environment.
Stage 4 Path Traversal
Vulnerability: The file browser concatenated user input directly into a file path without sanitization, allowing ../ sequences to read files outside the intended directory.
Fix: Sanitize file paths with os.path.basename() to strip directory components. Use an allowlist of permitted files. Validate that the resolved path stays within the intended directory.
